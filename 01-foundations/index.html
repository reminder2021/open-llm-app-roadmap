Python 与 Web API 基础

LLM 应用开发不是只会调一个模型接口。你需要能处理数据、设计 API、校验输入输出、记录日志、处理超时和流式响应。这一阶段先把工程底盘打稳。

建议周期：1-2 周 产出：文本处理 FastAPI 服务 重点：能独立写 API，不依赖 LLM 框架

本阶段模块

学习目标

完成这一阶段后，你应该能独立写出一个小型后端服务，为后续接入开源模型、RAG、工具调用和微调服务做准备。

能写 Python 服务

熟悉虚拟环境、依赖安装、模块组织、类型标注、异常处理和基础测试。

能设计 Web API

理解请求、响应、状态码、鉴权、错误格式，以及如何用 FastAPI 暴露服务。

能处理结构化数据

会读写 JSON、CSV、YAML，用 Pydantic 对输入输出做校验。

能排查线上问题

会记录日志、耗时、请求 ID、错误栈，能定位一次 API 调用为什么失败。

Python 必备能力

不需要成为 Python 专家，但要掌握 LLM 应用开发中高频出现的能力。

语法与结构
  • 函数、类、模块导入。
  • 列表、字典、集合、生成式。
  • 异常处理：try/except/finally。
  • 上下文管理器：with open(...)。
类型与模型
  • 类型标注：str、list[str]、dict。
  • Optional、Literal、Enum。
  • dataclass 的使用场景。
  • Pydantic 的输入输出校验。
工程习惯
  • 依赖写入 requirements.txt。
  • 密钥放在环境变量，不写死到代码。
  • 函数只做一件清晰的事。
  • 为关键逻辑写最小测试。

数据处理基础

LLM 应用会不断在“文本、JSON、数据库记录、API 响应、模型输出”之间转换。数据处理能力会直接影响系统稳定性。

必须会的格式

  • JSON：API 请求响应、模型结构化输出、工具参数。
  • CSV：批量样本、评估集、标注数据。
  • YAML：配置文件、实验参数、训练配置。
  • 纯文本和 Markdown：知识库、Prompt 模板、日志摘要。

练习任务

  1. 读取一个 Markdown 文件，统计字数、段落数和标题层级。
  2. 读取一个 CSV 文件，把每行转换成 JSON。
  3. 把一组文本样本保存成 jsonl，为后续微调数据格式做准备。
  4. 写一个函数，过滤空文本、超长文本和重复文本。

HTTP 与 API 基础

后续无论是调用模型、调用工具，还是把你的能力暴露给前端，本质上都离不开 HTTP API。

请求

  • GET：读取资源。
  • POST：提交任务或生成内容。
  • Header：鉴权、内容类型、请求 ID。
  • Body：JSON 请求体。

响应

  • 200：成功。
  • 400：请求参数错误。
  • 401/403：未登录或无权限。
  • 429：限流。
  • 500：服务端错误。

LLM 场景下的 API 设计要点

  • 每个请求都应该有唯一 request_id，方便排查问题。
  • 耗时任务要考虑超时、取消和异步执行。
  • 错误响应要稳定，不要把内部异常直接暴露给用户。
  • 模型输出不可信，进入下一步前要校验。
  • API 返回结构尽量固定，方便前端和评估脚本使用。

FastAPI 必会内容

FastAPI 很适合 LLM 应用原型和中小型后端服务。它和 Pydantic 的结合能帮你减少大量参数校验代码。

最小服务示例

from fastapi import FastAPI
from pydantic import BaseModel, Field

app = FastAPI(title="Text Processor API")

class TextRequest(BaseModel):
    text: str = Field(min_length=1, max_length=8000)
    language: str = "zh"

class TextResponse(BaseModel):
    summary: str
    keywords: list[str]
    length: int

@app.post("/process", response_model=TextResponse)
def process_text(payload: TextRequest) -> TextResponse:
    text = payload.text.strip()
    keywords = list(dict.fromkeys(text.replace("，", " ").replace("。", " ").split()))[:5]
    return TextResponse(
        summary=text[:80],
        keywords=keywords,
        length=len(text),
    )

需要掌握

  • 路由：@app.get、@app.post。
  • 请求模型和响应模型。
  • 路径参数、查询参数、请求体。
  • 统一异常处理。
  • 自动生成的 Swagger 文档：/docs。

异步、超时与流式响应

LLM 调用通常耗时较长，后续做流式输出、并发请求、工具调用时，异步能力会非常有用。

你需要理解

  • async def 和 await 的基本语义。
  • 用 httpx.AsyncClient 调用外部 API。
  • 为外部请求设置超时时间。
  • 用 FastAPI 返回流式响应。
  • 并发不是无限开，必须有限流和资源控制。
先能写同步版本，再改成异步版本。异步代码如果没有清晰边界，会让错误更难排查。

日志、配置与错误处理

LLM 应用很容易出现“偶尔答错、偶尔超时、偶尔格式坏掉”。没有日志，问题很难复现。

建议记录的字段

  • request_id：一次请求的唯一标识。
  • endpoint：调用了哪个接口。
  • latency_ms：耗时。
  • status：成功、参数错误、外部服务失败、内部异常。
  • input_length 和 output_length：避免记录完整敏感文本。
  • error_type：便于统计问题类型。

配置管理

  • 把端口、模型地址、API Key、超时时间放到环境变量。
  • 提供 .env.example，不要提交真实密钥。
  • 开发环境和生产环境使用不同配置。

两周练习安排

第 1-2 天：Python 项目结构

搭建虚拟环境，创建模块目录，完成文本读写、JSON/CSV 处理和异常处理练习。

第 3-4 天：Pydantic 与数据校验

为文本处理任务设计请求模型、响应模型和错误模型，练习字段长度、枚举值、默认值。

第 5-7 天：FastAPI 服务

实现 /health、/process、/batch-process 三个接口，并通过 /docs 测试。

第 8-10 天：异步与外部调用

用 httpx 调用一个公开 API 或本地假服务，加入超时、重试和错误处理。

第 11-14 天：项目整理

补充日志、README、测试用例和运行脚本，为下一阶段接入本地开源模型做准备。

实践项目：文本处理 API

做一个“文本处理 API”：接收一段文本，返回摘要、关键词、文本统计和结构化 JSON。先用规则实现，下一阶段再把规则处理替换为开源模型调用。

接口建议

  • GET /health：服务健康检查。
  • POST /process：处理单条文本。
  • POST /batch-process：批量处理多条文本。
  • GET /version：返回服务版本和配置摘要。

返回字段建议

  • summary：短摘要。
  • keywords：关键词列表。
  • stats：字符数、段落数、句子数。
  • warnings：输入过短、过长、疑似为空等提示。
  • request_id：请求 ID。

验收标准

  • 能用 uvicorn app.main:app --reload 启动。
  • 请求参数有 Pydantic 校验。
  • 错误时返回稳定 JSON，而不是直接抛异常页面。
  • 日志里能看到请求 ID、接口、耗时和错误类型。
  • 至少有 5 个测试用例：正常文本、空文本、超长文本、批量文本、非法参数。
  • README 能说明如何安装、启动、调用接口。

推荐资料
